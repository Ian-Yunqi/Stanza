🔴 **未决策的问题：**

1. **标签点击后，是一次推荐一本还是展示多本？**
   - 方案 A：点标签 → 直接出"处方"（一本），干净利落，用完即走。想看另一本必须返回重选或点"换一剂"
   - 方案 B：点标签 → 出现一个极简书名列表（2-3 本），用户再选一本进入处方
   - **建议：方案 A。** 符合"药方"隐喻——你不会去药房让药剂师给你一排药自己挑。如果要换，底部给一个极小的"换一剂"文字链

2. **"每日一剂"和标签之间的关系是什么？**
   - 每日一剂是否也显示在"处方页"，结构完全一样？→ **应该是，只是入口不同**
   - 每日一剂选书逻辑：完全随机还是优先当天没被用户点过的标签？

3. **用户点过标签后回到首页，需要标记"已服用"吗？**
   - 不需要——"不用黏住"意味着不记录用户行为
   - 但"每日一剂"需要 localStorage 记录当天是否已查看（见下文逻辑层）

### 页面 2：处方页（导读）

```
┌──────────────────────────────┐
│ ←                            │  ← 极轻的返回箭头，左上角
│                              │
│                              │
│   002 · 给青年的十二封信      │  ← 编号 + 书名，小号字体
│   ──────────────────────     │
│                              │
│   你并不是不知道自己          │
│   要什么。你知道得            │
│   太多了。                   │
│                              │
│   [200-400 字认知重塑导读]    │  ← 核心内容区
│   ...                        │
│   ...                        │
│                              │
│                              │
│                              │
│            ↓                 │  ← 半透明，可能带极轻的呼吸动画
│       （下滑试读）            │  ← 超小字体，可选
│                              │
└──────────────────────────────┘
```

🟡 **优化建议：**

1. **编号（001、002...）要不要显示？**
   - 显示。给用户一种"这是一套体系"的暗示，增强信任感。即使选了"无账号"，编号也暗示"你只是这个序列中的一个人"，增加仪式感但不增加负担
   
2. **书名位置：前置还是后置？**
   - 目前放在导读前面 → **建议放在导读后面**（金句区的开头），原因：先用药，后知药名。这是"书方"区别于书评网站的关键——用户因为情绪而来，不是因为书而来。书名先出现会产生"哦这本我听过/没听过"的预判，干扰导读的认知重塑效果

3. **返回箭头的位置和样式**
   - 左上角，极轻（rgba 20% 黑），不抢注意力
   - 也可以用"← 换一剂"替代纯返回，暗示"不满意可以换"

```
│   提示：前两封稍远，撑到第三封 │

│                              │
└──────────────────────────────┘
```

🔴 **未决策的问题：**

1. **金句的逐条露出 vs 一次性展示？**
   - 需求文档说"下滑箭头 ↓ 直接展示高光段落"
   - 但 3-5 条金句如果同时出现，会失去节奏感 → **建议逐条露出**：用户下滑时，每滑过一个屏幕高度出现一条新金句。利用 Intersection Observer + 淡入动画，极克制（opacity 0→1，无位移，时长 0.6s ease）

2. **金句之间用什么分隔？**
   - 不能是花哨的分割线 → **一条极细的横线（1px, #eee 或更淡），金句之间的距离大于导读段落间距的 2 倍**，让每一条都有"呼吸"

3. **金句来源要不要标注？**
   - 需求文档说"借鉴各大平台标注最多的句子" → **不标注来源平台和人数**。这些数字在阅读场景里是噪音。金句的力量来自文字本身，不是来自"很多人也喜欢"

## 二、交互逻辑层

### 2.1 路由设计

```
/                    → 首页（情绪标签 + 每日一剂）
/?tag=迷茫           → 用"迷茫"标签匹配一本书 → 处方页
/?dose=today         → 每日一剂 → 处方页
/?tag=迷茫&book=003  → 指定书籍（换一剂跳转时使用）
```

🟡 所有参数放在 URL query 而非 path 中，因为不需要 SEO，且方便分享（虽然不需要鼓励分享，但不应该阻止

### 2.2 标签 → 书籍匹配算法

```
输入：用户点击的标签 T
输出：一本书 B

步骤：
1. 从书库中筛选"主标签 == T"的书 → 候选池 P1
2. 如果 P1 为空，筛选"辅标签 == T"的书 → 候选池 P2
3. 从 P1（或 P2）中排除 localStorage 中记录的"最近 5 本已看"
4. 如果排除后为空，回到未排除的池子
5. 加权随机选一本（权重可调：短篇优先？评分优先？暂时等权）
6. 将选中书 ID 推入 localStorage "最近已看"队列（保持 ≤ 5 条）
```

🔴 **需要决策：** 同一标签反复点击 → 是否总是随机换一本？还是记住上一次的推荐？→ **建议随机换一本**，因为"换一剂"的隐喻就是每次抓一副新药。如果用户想看上次那一本，那是书架类产品的事，不是书方的。这是一个 feature，不是一个 bug。

### 2.3 每日一剂逻辑

```
输入：当天日期 D
输出：一本书 B

步骤：
1. 读取 localStorage key: "daily-dose-YYYY-MM-DD"
2. 如果存在 → 返回已记录的 book ID（同一天多次打开，看到同一本书）
3. 如果不存在：
   a. 从全部 30 本书中随机选一本
   b. 存入 localStorage "daily-dose-YYYY-MM-DD" = book ID
   c. 返回 book ID
4. 清理：删除超过 7 天的旧 daily-dose key（防止 localStorage 膨胀）
5. 额外处理：如果今天的 daily book 恰好被用户通过标签也看过了 → 不用处理，"换一剂"已经解决了重复问题
```

🟡 **"每日一剂"的刷新时间：** 凌晨 0 点（本地时间）。JS 直接用 `new Date().toLocaleDateString('zh-CN')` 做 key，自然在午夜切换。

### 2.4 localStorage 设计

```
Key                          Value           说明
───────────────────────────────────────────────────
daily-dose-2026-06-05        "003"           每日一剂记录（7 天自动清理）
seen-books                   ["003","001"]   最近 5 本已看（FIFO 队列）
```

仅此两条。不存任何用户偏好、历史、统计。

🟢 这保持了"无账号"承诺——localStorage 是浏览器本地行为，用户清除浏览器数据即归零，不存在服务端痕迹。

### 2.5 交互细节决策

| 场景 | 决策 |
|------|------|
| 标签点击 | 直接进入处方页（一本书），无中间步骤 |
| 处方页看到一半想换 | 底部"换一剂"或左上角 ← 回到首页 |
| 金句读完后 | 底部"换一剂"（同标签换另一本）+ 极小的"回到首页" |
| 冷启动（书库为空） | 不适用——书籍数据静态打包 |
| 网络断开 | 全部静态页面，首次加载后完全离线可用 |
| 分享链接 | 不做任何分享按钮。URL 可以复制，但页面不鼓励 |
| 深色模式 | 第一版不做。极简留白依赖白色背景 |
| 字体大小 | 使用系统默认 + rem，尊重用户无障碍设置 |

## 三、视觉规范补充

需求文档已确定"极简留白 + 纯文字向"。以下是未明确但应该统一的：

| 要素 | 建议 |
|------|------|
| 字体 | 系统默认中文字体栈：`"PingFang SC", "Noto Serif SC", "Source Han Serif SC", serif` — 导读和金句用衬线体，情绪标签用无衬线（视觉区分"选择"和"阅读"两种模式） |
| 背景色 | `#fafaf9`（暖白，非纯白 #fff — 纯白在 OLED 屏幕上刺眼） |
| 文字色 | `#1a1a1a`（非纯黑 #000 — 纯黑和白底对比度过高） |
| 正文字号 | 导读 16px / 行高 2.0，金句 18px / 行高 2.2（金句略大，暗示"这是重点"） |
| 标签字号 | 14px，字间距 0.05em（中文字不需要太大，靠留白撑气场） |
| 页面最大宽度 | 640px，居中（防止大屏手机上行宽过宽影响阅读） |
| 左右内边距 | 32px（移动端） / 48px（≥640px 时） |
| 箭头 ↓ | 半透明 rgba(0,0,0,0.15)，无文字时 24px，有"下滑试读"文字时 20px |
| 动画 | 仅有页面切换的淡入（opacity 0→1, 400ms ease）和箭头呼吸（opacity 0.1→0.3, 2s 循环）。**不用任何滑动、弹跳、缩放** |

## 四、完整交互流程（终版）

```
用户打开 / 
  │
  ├─ 看到标题"书方"，8 个情绪标签，以及下方"今日一剂"
  │
  ├─ 点击「迷茫」
  │     │
  │     ├─ 1. 匹配算法选书
  │     ├─ 2. 推入 seen-books 队列
  │     ├─ 3. 进入处方页 /?tag=迷茫
  │     │
  │     ├─ 看到：认知重塑导读（书名不出现）
  │     ├─ 看到：底部 ↓ 箭头，轻微呼吸动画
  │     │
  │     ├─ 下滑（或点箭头）
  │     │     │
  │     │     ├─ 平滑滚动到金句区
  │     │     ├─ 第 1 条金句淡入
  │     │     ├─ 继续下滑 → 第 2、3、4 条逐条淡入
  │     │     ├─ 最后：书名、获取方式、相似推荐（极小字）
  │     │     ├─ 底部："换一剂 →"（同标签换一本） / "回到首页"
  │     │     │
  │     │     └─ 点"换一剂" → /?tag=迷茫 → 重新匹配一本书
  │     │
  │     └─ 点 ← 返回首页
  │
  ├─ 点击「今日一剂」
  │     │
  │     ├─ 1. 检查 localStorage 今日是否已有记录
  │     ├─ 2. 有 → 显示今天那本 / 无 → 随机选一本并记录
  │     ├─ 3. 进入处方页 /?dose=today
  │     └─ 后续同标签流程（导读 → 金句 → 换一剂/返回首页）
  │
  └─ "换一剂"逻辑
        │
        ├─ 如果是标签入口 → 同标签换一本
        └─ 如果是每日一剂入口 → 全库随机换一本（今天每日一剂记录不更新）
```

## 五、发现的问题 & 边界情况

| # | 问题 | 严重度 | 处理方案 |
|---|------|--------|----------|
| 1 | 书只有 30 本，同一标签只有 3-4 本。用户连续点击 4 次同一标签后，可能看完了该标签所有主标签书 | 🟡 中 | 第 5 次点击时，算法自动降级到辅标签匹配，并在顶部极小字提示"本日迷茫之药已尽，以下为相近之方" |
| 2 | 每日一剂和标签推荐可能撞书 | 🟢 低 | 不做特殊处理。用户通过标签看到一本书和通过每日一剂看到同一本书，导读一样，但入口不同，心境不同——这不是 bug |
| 3 | 浏览器清除 localStorage 后，"每日一剂"会在同一天变一本新书 | 🟢 低 | 符合预期。清除浏览器数据等于清除所有痕迹，产品重新变成一个陌生人。不视为 bug |
| 4 | 微信内置浏览器可能禁用 localStorage | 🟡 中 | 需要降级处理：用 cookie 或 sessionStorage 做 fallback。所有 localStorage 操作包裹 try-catch。如果全部不可用，"每日一剂"在同一天内退化为"每次随机一本" |
| 5 | 金句区很长（5 条金句 + 延伸信息），用户可能在导读下滑后失去终点感 | 🟡 低 | 最后一条金句之后加一个极小的视觉终止符，如"□"（空心方框），暗示"药服完了"。这是印刷品的古典终章标记，符合书方的调性 |
| 6 | 导读 200-400 字，在 375px 宽屏幕上约 20-40 行。如果导读太长，↓ 箭头可能被挤到首屏之外 | 🟡 中 | 导读超过 300 字时，↓ 箭头始终固定在视口底部（position: fixed, bottom: 24px），而非跟随文档流 |
| 7 | 换一剂后 URL 改变，但没有浏览历史管理——用户可能通过浏览器返回按钮回到上一本书 | 🟢 低 | 不做特殊处理。浏览器原生 back 行为是合理的，且符合"不黏住"哲学 |
