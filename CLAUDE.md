## 项目概述

你要开发的是一款名为"书方"的移动端 H5 网页产品。

**一句话定义：** 专治大学生"无聊、迷茫、厌倦"等情绪的精神处方。不是书评网站，不是推荐引擎，是一台"情绪无人售药机"。

**核心哲学：** 极简、高维认知重塑、无账号、无压力、用完即走。

## 产品核心规则（开发前必须理解）

> 以下规则决定"书方"为什么是"书方"而不是普通书评站。违反任何一条，产品灵魂就变了。

1. **标签点击 → 直接出导读（一本书），不展示列表** — 药剂师不给你一排药自己挑
2. **导读区不出现书名和作者** — 先用药，后知药名。书名在金句区才首次出现
3. **金句逐条淡入（IntersectionObserver）** — 3-5 条金句依次露出，不是一次性堆出来
4. **↓ 箭头固定在视口底部（position: fixed）** — 防止长导读把箭头挤出首屏
5. **视觉规范** — 暖白背景（#fafaf9）、衬线字体、页面最大宽 640px、纯文字向
6. **无账号、不黏住** — 仅用 localStorage 记录"最近 5 本已看"和"今日一剂"，不存任何用户数据

## 页面结构（一共 2 个页面）

```
首页 (/)               处方页 (/?tag=迷茫)
─────                  ─────────────────
"书方"                 认知重塑导读（200-400字）
                       无书名、无作者
8 个情绪标签             ↓ 箭头（固定底部，呼吸动画）
（无聊 迷茫 厌倦 浮躁   ─────────────────
 疲惫 空游无所依        第 1 条金句（淡入）
 烦闷 平淡）           第 2 条金句
                       第 3-5 条金句
每日一剂                （书名在此处首次出现）
（更小字号）            延伸信息（极小字）
                       □（终止符）
                       "换一剂 →" / "回到首页"
```

## 技术栈一句话

**纯静态 H5：1 个 HTML + 1 个 CSS + 1 个 JS + 1 个 books.json。** 无框架、无后端、无数据库。AI（Gemini 3.1 Pro）只在构建阶段生成导读内容，用户访问时零 API 调用。部署到 Cloudflare Pages。

## 特别提醒

- 书名和作者在**导读区不出现**，只在金句区开头首次出现。这是"书方"区别于书评网站的核心 UX 决策
- 金句逐条淡入用 `IntersectionObserver`，不是 `scroll` 事件
- ↓ 箭头是 `position: fixed`，不是文档流
- 情绪标签字体用无衬线 → 导读/金句用衬线。视觉区分"选择"和"阅读"两种模式
- 不装任何分析工具（GA、百度统计等），不设任何分享按钮。"用完即走"不是口号

## AI 协作与工作流守则

> **本段规则约束一切在此项目中工作的 AI（包括 Claude、Gemini 及任何辅助编码 Agent）。**

### 1. 强制性文档查阅顺序

在开始任何开发任务之前，AI **必须**按优先级依次查阅以下文档，不得跳过：

| 优先级 | 文件名 | 查阅目的 |
|--------|--------|----------|
| P0 | `CLAUDE.md`（本文件） | 理解项目全貌、交付文件清单、核心规则 |
| P1 | `需求文档.txt` | 理解产品哲学与所有设计决策的"为什么" |
| P2 | `UI与逻辑审查.md` | 掌握交互流程、状态机、匹配算法、视觉规范 |
| P3 | `技术选型与开发方案.md` | 理解技术栈、架构、部署方式 |
| P4 | `书籍案例库.md` | 了解已有数据格式与内容调性 |
| P5 | `种子书库.md` | 了解标签映射与书库覆盖矩阵 |
| P6 | `样书案例模板.md` | 新增书籍时参照此模板 |

**未读完上述文档之前，不得开始编写任何代码。**

### 2. 需求与设定类文档保护规则

以下文件属于"需求与设定类 Markdown 文档"，**严禁 AI 擅自修改**：

- `UI与逻辑审查.md`
- `需求文档.txt`
- `技术选型与开发方案.md`
- `种子书库.md`
- `样书案例模板.md`
- `书籍案例库.md`
- `CLAUDE.md`

**AI 对这些文档的操作权限：只读。** 如需修改上述任何文件，必须：
1. 明确向用户说明修改理由与具体变更内容
2. 获得用户明确同意后，方可执行修改

违反此条规则的后果：产品的设计意图与实现之间将产生不可逆的偏离。

### 3. 复杂修改前的方案输出机制

对 `书方.html` 或项目核心架构实施以下操作时，AI **必须先输出方案供用户确认，再执行修改**：

- 修改交互逻辑（页面跳转、路由、状态管理）
- 修改核心 UI 规则中定义的 6 条强制规则
- 新增或删除情绪标签
- 大规模重构 CSS 布局结构
- 修改书籍匹配算法（`matchBook` 函数）
- 修改数据持久化逻辑（`localStorage` 键名、数据结构）

**方案输出格式：**
1. 当前问题 / 需求描述
2. 拟修改的文件与位置
3. 修改方案（变更前后对比）
4. 潜在影响分析
5. 等待用户回复"可以"或"执行"后再动手

### 4. 日常工作模式

- **简单修改**（错别字、CSS 数值微调、单参数调整）：可直接执行
- **新增书籍数据**：参照 `样书案例模板.md` 格式，写入 `书方.html` 的 `BOOKS` 数组。不得修改已有 030 条之前的任意书籍数据条目
- **代码风格**：使用 `var`（非 `let`/`const`）保持与现有代码一致；用 `function` 声明而非箭头函数以兼容老式浏览器
- **提交/部署**：未经用户明确指示，不执行 `git commit`、`git push` 或任何部署操作

### 5. 质量守则

- 所有修改完成后，必须简要汇报：改了什么、为什么这样改、潜在风险是什么
- 如果用户提出了一个与现有产品规则冲突的需求，AI 应先指出冲突，而不是默默执行
- 测试只在用户明确要求时执行。不主动运行政务相关的命令
