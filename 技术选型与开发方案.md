# "书方"技术选型与开发方案

> 基于产品需求 + Gemini 3.1 Pro + 纯静态 H5 的极简技术架构。
> 目标：零服务器成本、零运维、零依赖。一个域名 + 一套静态文件 = 完整产品。

---

## 一、总览：极简三层

```
┌──────────────────────────────────────────────┐
│                   用户                        │
│              (移动端浏览器)                    │
└──────────────────┬───────────────────────────┘
                   │ HTTPS
┌──────────────────▼───────────────────────────┐
│           CDN (Vercel / Cloudflare Pages)     │
│                                              │
│   /index.html        — 首页                   │
│   /dose.html         — 处方页（导读+金句）     │
│   /data/books.json   — 全部书籍数据            │
│   /app.js            — 路由 + 匹配 + 渲染      │
│   /style.css         — 极简留白样式            │
│                                              │
│   ⚡ 首次加载后全部缓存，离线可用               │
└──────────────────────────────────────────────┘
                   │
                   │ (仅在"生成内容"时触发)
┌──────────────────▼───────────────────────────┐
│          Gemini 3.1 Pro API                   │
│                                              │
│   用途：批量生成 认知重塑导读                   │
│   触发：开发阶段一次性调用                     │
│   频率：新增书籍时手动运行一次                  │
│                                              │
│   ❌ 不暴露给浏览器                            │
│   ❌ 不在运行时调用                            │
│   ✅ 结果写入 books.json，静态部署              │
└──────────────────────────────────────────────┘
```

核心决策：**AI 只在构建阶段工作，不在运行时工作**。用户打开网页时，读的是预生成好的静态 JSON，零延迟、零 API 费用、零密钥泄露风险。

---

## 二、为什么用这个架构

| 问题 | 方案 A（运行时调 AI） | 方案 B（构建时调 AI） | 选择 |
|------|----------------------|---------------------|------|
| 每次打开都调 Gemini？ | 是，烧钱 + 慢 | 不需要，数据已生成 | **B** |
| API Key 放哪？ | 必须放在前端 → 泄露 | 只在开发机本地 | **B** |
| 离线能不能用？ | 不能 | 能 | **B** |
| 响应速度 | 等 2-5 秒 | 即开即看 | **B** |
| 能动态更新吗？ | 可以实时生成 | 需要重新构建部署 | 接受 |
| 匹配算法怎么做？ | 先把用户情绪总结再搜书 | 标签直接映射，不需要 LLM | **标签匹配** |

"书方"的全部"AI 感"都在**内容**里（导读、金句），不在**交互**里。交互是纯规则引擎。所以 AI 只负责产出内容，不负责服务用户。

## 三、Gemini 3.1 Pro 使用方案

### 3.1 在哪里用

| 场景 | 用什么 | 频率 |
|------|--------|------|
| 生成"认知重塑导读" | Gemini 3.1 Pro | 每本新书一次 |
| 筛选"金句" | 人工 + Gemini 辅助润色 | 每本新书一次 |
| 给新书分配标签 | 人工判断 | 每本新书一次 |
| 匹配算法（运行时） | 纯规则，不用 AI | — |

### 3.2 导读生成脚本（Node.js，本地运行）

```js
// scripts/generate-dose.mjs
// 用途：给一本新书生成认知重塑导读
// 用法：node scripts/generate-dose.mjs --book=books/031.json
// 前提：book.json 中已填写基本信息、标签、金句

import { GoogleGenAI } from "@google/genai";
import { readFileSync, writeFileSync } from "fs";

const ai = new GoogleGenAI({ apiKey: process.env.GEMINI_API_KEY });

const book = JSON.parse(readFileSync(process.argv[3], "utf-8"));

const prompt = `你是一个名为"书方"的精神向导。你的任务是为一本具体的书撰写"认知重塑导读"。

## 你的定位
- 你不是书评人，你是精神引路人。
- 你不提供廉价共情，你提供高一个维度的认知视角。
- 你的读者是深陷碎片化信息、注意力涣散、情绪漂浮但内心渴望深度阅读的大学生。

## 你的写作要求
${readFileSync("prompts/guide-prompt.md", "utf-8")}

## 这本书的信息
- 书名：${book.title}
- 作者：${book.author}
- 一句话简介：${book.summary}
- 匹配的情绪标签：${book.tags.join("、")}
- 代表性金句：
${book.quotes.map((q, i) => `${i + 1}. ${q.text}`).join("\n")}

现在请为这本书撰写认知重塑导读。`;

const response = await ai.models.generateContent({
  model: "gemini-2.5-pro",  // 用户实际使用 gemini-3.1-pro 时替换
  contents: prompt,
});

book.guide = response.text;
writeFileSync(process.argv[3], JSON.stringify(book, null, 2), "utf-8");
console.log(`✅ 导读已写入 ${process.argv[3]}`);
```

### 3.3 为什么要本地脚本而不是在线调

- API Key 永远不离开开发机
- 生成结果可以人工校对后再部署（导读质量 > 速度）
- 一次性成本：30 本书 × 约 500 tokens/本 ≈ 15k tokens ≈ $0.05
- 如果部署后实时调 API：每次匹配都调 → 钱烧得快 + 用户等得久 + Key 暴漏风险

## 四、前端技术选型

### 4.1 框架：不用框架

| 候选 | 理由 | 决策 |
|------|------|------|
| React/Vue | 2 个页面、零状态管理、零组件树 → 杀鸡用牛刀 | ❌ |
| Svelte | 轻量，但仍然要多一层编译 | ❌ |
| Alpine.js | 极轻（15KB），声明式交互 | 🟡 可考虑 |
| 原生 HTML + JS | 完全掌控、零依赖、文件体积最小 | ✅ **首选** |

**结论：一个 HTML 文件 + 一个 CSS 文件 + 一个 JS 文件。总大小 < 50KB（不含书籍数据）。**

### 4.3 关键 JS 实现要点

**路由（router.js）**


**匹配算法（matcher.js）**
```js
function matchBook(tag, books, seenIds) {
  // 1. 主标签 == tag 的候选池
  let pool = books.filter(b => b.tags.primary === tag);
  // 2. 为空则降级到辅标签
  if (!pool.length) pool = books.filter(b => b.tags.secondary.includes(tag));
  // 3. 排除最近5本已看
  const fresh = pool.filter(b => !seenIds.includes(b.id));
  // 4. 如果全部看过，回到未排除池
  const candidates = fresh.length ? fresh : pool;
  // 5. 加权随机
  return candidates[Math.floor(Math.random() * candidates.length)];
}
```

## 五、部署方案

### 5.1 推荐：Cloudflare Pages

| 维度 | Cloudflare Pages | Vercel | GitHub Pages |
|------|-----------------|-------|-------------|
| 免费额度 | 无限带宽 | 100GB/月 | 100GB/月 |
| 国内访问速度 | 🟡 中等 | 🟡 中等 | 🟡 中等 |
| 自定义域名 | ✅ | ✅ | ✅ |
| 自动 HTTPS | ✅ | ✅ | ✅ |
| 构建命令 | 不需要（纯静态） | 不需要 | 不需要 |
| 部署方式 | `wrangler pages deploy` 或 Git 推送 | Git 推送 | Git 推送 |

**建议：Cloudflare Pages**。纯静态文件，直接上传。不需要 Git 集成就拖拽部署。

### 5.2 部署步骤

```bash
# 1. 安装 wrangler（一次性）
npm install -g wrangler

# 2. 登录
wrangler login

# 3. 部署（在 shufang/ 目录下）
wrangler pages deploy . --project-name=shufang

# 4. 绑定自定义域名（在 Cloudflare 控制台操作）
# 结果：https://shufang.你的域名.com
```

或者更简单：把文件夹拖进 Cloudflare Pages 网页控制台。

## 八、特别说明：Gemini 3.1 Pro vs 其他模型

| 场景 | Gemini 3.1 Pro 优势 | 注意 |
|------|-------------------|------|
| 生成导读 | 中文极好，自带"克制感"——不容易写成鸡汤 | 需要明确 prompt 里的"不用感叹号""不写废话"约束 |
| 长上下文 | 1M token 窗口 → 可以一次喂入多本书的信息批量生成 | 批处理时每本书的 prompt 要分开，避免串味 |
| 成本 | 极其便宜（约 $1.25/1M input tokens） | 30 本书的总成本 < $0.10 |
| 速度 | 快 | 适合本地脚本一次性调用 |

**如果不想自己跑脚本**：也可以直接在 Gemini 网页版对话里，把每本书的信息贴进去，复制输出。30 本书手动做也不超过两小时。不需要写代码，不需要配 API Key。

## 九、不需要做的事

| 不要做的 | 原因 |
|----------|------|
| 数据库（MySQL/Postgres/Supabase） | 数据是 30 本书的 JSON，静态文件就够了 |
| 后端框架（Express/FastAPI） | 没有用户系统、没有实时计算、没有需要保护的数据 |
| 用户系统/登录 | "不用黏住"是产品原则 |
| 推荐算法/协同过滤 | 标签匹配 + 随机就够了——"药方"不需要猜你喜欢什么 |
| 前端框架 | 两个页面，原生 JS 更轻更快 |
| CI/CD 流水线 | 纯静态，改了文件拖上去就行 |
| 容器化/Docker | 没有后端，没有服务 |
| 监控/日志/分析 | "不用黏住"——不收集用户数据 |
